# Inkhaven 1.2.14 — in flight

1.2.13 was the **Language book** release — a
first-class invented-language workbench with
five-chapter sub-books (Meta / Dictionary /
Grammar / Phonology / Sample texts), HJSON-
fronted entries, lexicon overlay + status chip
in the manuscript editor, AI translation flow
with `Ctrl+B Q` / `Ctrl+B Shift+Q`, full CLI
surface (init, add-word, --import CSV with --new
/ --force, remove-word, list, doctor with --json,
export), and two tutorials (49 + 50) with a
worked Tira sample CSV.

This document is a living scratchpad — entries
land here as the work lands.  Each item starts
as a one-line plan; gets fleshed out when the
corresponding feature ships; gets a chord /
config / hook callout when relevant.

### Carryovers from 1.2.13

#### Language book — Phase D.2 (deferred from 1.2.13)

  * **`--format grammar` export** — printable
    grammar reference (TOC + chapter per rule
    category + examples tables + cross-refs).
    The current grammar HJSON template carries
    the right shape; the exporter needs to
    parse + format it.
  * **`--format phrasebook` export** — Sample-
    text chapter rendered as a phrasebook,
    gloss on left, working language on right.
  * **`--format csv` export** — round-trip-
    compatible CSV that the `--import` path
    can re-ingest.  Closes the import/export
    loop.
  * **`inkhaven language test <name>`** —
    headless roundtrip translation drift CLI.
    Walks a corpus of sample texts through
    `Ctrl+B Q` → `Ctrl+B Shift+Q` and reports
    drift.  Catches grammar-rule inconsistencies
    before they bite in the manuscript.  The
    in-TUI version of this workflow already
    works; this CLI version is for CI gates.
  * **`inkhaven language translate <language>
    <text>`** — headless flavour of the in-TUI
    `Ctrl+B Q` chord.  Useful for shell-pipe-
    driven authoring.
  * **`inkhaven language define-rule <language>
    <rule_id> --category <cat>`** — open the
    rule template in `$EDITOR` for hand-editing
    the HJSON.  Pairs with the deferred
    `--format grammar` exporter.

## Implementation log

### Threads system book — Phase A.1 (foundation)

Lays the groundwork for the plot-intelligence
tooling described in
`Documentation/PROPOSALS/1.2.14_PLAN.md`.  Phase
A.1 is the storage + scaffold layer only — no
weave view (A.2), no AI thread audit (A.3).

* **`SYSTEM_TAG_THREADS = "threads"`** added to
  `src/store/mod.rs`.  Top-level `Threads` system
  book auto-creates on next project open (or
  upgrade from a pre-1.2.14 project) alongside the
  existing system books.  Lands at canonical order
  index 6 — after Artefacts, before Language — so
  the worldbuilding cluster (Places / Characters /
  Artefacts / Threads / Language) groups together
  in the tree.  Existing user books shift up by 1
  if they collided with the slot; the heal-pass
  at the end of `ensure_system_books` mops up.

* **`inkhaven thread add <name>`** CLI subcommand.
  Creates a Paragraph under the Threads system
  book, seeds the body with the full commented
  HJSON template per proposal §3.1 (identity:
  `title` / `status` / `weight` pre-filled from
  the CLI flags; arc shape: `opening` /
  `midpoint` / `payoff`; connections:
  `characters` / `places` / `artefacts` /
  `related_threads`; metadata: `tension` /
  `register` / `notes`).  Sets `content_type =
  "hjson"` so the editor renders with syntax
  highlighting + the paragraph status bar shows
  `[hjson]`.  Disk-write + bdslib-write follow
  the canonical two-step pattern from Language
  entries.  Rejects duplicates by case-insensitive
  title.

  ```
  $ inkhaven thread add "inheritance subplot" \
      --status develop --weight subplot
  added thread `inheritance subplot` to Threads (develop · subplot)
    open Threads/inheritance subplot in the editor to fill opening / midpoint / payoff
  ```

* **`inkhaven thread list`** CLI subcommand.
  Walks the Threads subtree, parses each entry's
  HJSON summary fields, prints a column-aligned
  table:

  ```
  $ inkhaven thread list --status develop
    name                  status    weight  tension   ch  pl  art
    ----------------------------------------------------------
    inheritance subplot   develop  subplot        7    2   1    0
    redemption arc        develop    major        5    1   0    1
  ```

  `--status` / `--weight` flags filter case-
  insensitively.

* **Tree-pane `+` chord under Threads** —
  `commit_add` now detects when the parent's
  ancestor chain includes the Threads system book
  and seeds the body with the thread HJSON
  template (`title` pre-filled from the typed
  name; `status: "setup"`, `weight: "major"`).
  `content_type` flips to `"hjson"`.  Same
  template-seed pattern Language Grammar/Phonology
  paragraphs use, just with a different schema.
  No bucket-routing needed — threads are flat
  Paragraphs under the system book (or chapters
  the author adds for grouping).

* **Phase A.1 tests** — 7 new:
  `seed_thread_body_includes_core_fields`,
  `seed_thread_body_uses_name_when_title_omitted`,
  `seed_thread_body_prefers_explicit_title`,
  `seed_thread_body_parses_as_valid_hjson`,
  `parse_thread_summary_extracts_filter_fields`,
  `parse_thread_summary_returns_none_on_empty`,
  `escape_hjson_handles_quotes_and_backslashes`.

572 tests pass (was 565 — +7).  Zero new
dependencies.  Phase A.2 (thread weave view +
`Ctrl+V Shift+H` picker) and Phase A.3 (AI thread
audit + `Ctrl+V Shift+A`) are queued.

### Threads — Phase A.2 (picker + weave view)

The first user-facing payoff of the Threads
system book.  Project-wide picker over every
plot-thread paragraph, with a swim-lane weave
sub-view that surfaces which chapters advance
which threads.

* **`Ctrl+V Shift+H`** opens the **Threads
  picker** (action: `view.threads_picker`).
  Centred modal listing every thread with
  columns: name (title field from HJSON;
  falls back to slug-derived paragraph title)
  · status · weight · tension · character
  link count · place link count ·
  reverse-link count (how many manuscript
  paragraphs link TO this thread via the
  existing `Ctrl+V A` outgoing-link
  mechanism).

  Picker chords:
  * `↑ ↓ Home End` — navigate the visible
    rows.
  * `Enter` — open the thread entry in the
    editor.
  * `Shift+Enter` — pin to the split-view
    secondary slot (mirrors the existing
    universal Shift+Enter pin modifier from
    1.2.12).
  * `/` — enter filter mode; typed substring
    matches against name + title + status +
    weight (case-insensitive).  `Enter` /
    `Esc` exit filter mode without closing the
    picker.
  * `w` — open the weave-view sub-modal.
  * `Esc` — close picker.

  Reverse-link counts are tallied in a single
  pass over the hierarchy at picker-open time
  so we don't pay `O(threads × paragraphs)`
  per render.

* **`w` from picker** opens the **swim-lane
  weave view** — a `Modal::ThreadWeaveView`
  pushed with the picker stored in
  `return_to` so `Esc` pops back.  Layout:

  ```
   ┌─ Thread weave · 4 threads × 12 chapters ──────┐
   │                man         man-ru             │
   │                 1  2  3  4  5  6  7  8  9 ...  │
   │ inheritance   · ●  ●  ·  ·  ●  ·  ·  ·  ●  ·   │
   │ secret-cult   ·  ·  ●  2  ·  3  ●  ·  ●  ·   │
   │ redemption    ●  ·  ·  ·  ●  ·  ●  ·  ·  ●   │
   │ comic-relief  ·  ·  ·  ·  ·  ·  ·  ●  ·  ·   │
   │                                                │
   │ inheritance subplot · manuscript-en/05-rain · 1 │
   │ ↑↓ thread · ←→ chapter · Enter jump · Esc back │
   └────────────────────────────────────────────────┘
  ```

  Cell content: `·` for zero links, `●` for
  one, the count for two or more.  Cursor
  cell is reversed; non-empty cells colour
  with `theme.places_fg` to match the
  manuscript overlay convention.  Book
  grouping shown in a header row above the
  chapter index numbers (3-letter book
  abbreviation, repeated only on book
  boundaries to avoid visual noise).

  Weave-view chords:
  * `↑ ↓` — move between threads.
  * `← →` — move between chapters.
  * `Home End` — jump to first / last
    chapter.
  * `Enter` — open the first linking
    paragraph in the cursor cell.  Status
    message when the cell is empty.
  * `Esc` — return to the picker (modal stack
    via `return_to: Box<Modal>`).

  Footer line tracks the cursor coordinate:
  `<thread title> · <book>/<chapter> · N
  linking paragraphs`.

  Grid pre-computed at modal-open time so
  navigation is pure cursor math; no per-
  keystroke hierarchy walks.

* **Surfaces:**
  * `Action::ViewThreadsPicker` — rebindable
    via the standard `keys.bindings` HJSON
    chord-customisation channel.
  * `Modal::ThreadsPicker` + `Modal::ThreadWeaveView`
    + `ThreadsPickerEntry` struct in
    `src/tui/modal.rs`.
  * `src/tui/app/threads_impl.rs` — new
    sub-module of `tui::app` carrying the
    entry-collection + reverse-link tally +
    picker handler + weave handler.
  * `draw_threads_picker_modal` +
    `draw_thread_weave_modal` in
    `src/tui/app/render/modals.rs`.

* **Phase A.2 tests** — 4 new pinning the
  new `truncate_to` Unicode-safe helper:
  `keeps_short_strings_intact`,
  `truncates_with_ellipsis`,
  `handles_max_one_or_zero`,
  `unicode_safe_no_byte_split` (Cyrillic 2-byte
  chars).

dependencies.  Phase A.3 (AI thread audit +
`Ctrl+V Shift+A`) queued.

### Threads — Phase A.3 (AI thread audit)

The headline AI feature of the plot-intelligence
theme.  Reads every Thread paragraph's HJSON +
runs a structural blind-spots pre-pass + walks
the cursor's scope, then asks the LLM to score
each scope paragraph against the thread
inventory and call out drift.

* **`Ctrl+V Shift+A`** (action:
  `ai.thread_audit`) — fires the audit from any
  pane.

* **Scope resolution from the F9 AiMode chip:**
  * `AiMode::Book` / `Chapter` / `Subchapter`
    → walk up from the cursor (open paragraph
    when there is one, otherwise the tree-pane
    cursor) until a node of that kind is found.
  * `AiMode::Paragraph` / `Selection` / `None`
    → fall back to the cursor's containing
    Chapter.  Auditing a single paragraph is too
    narrow to be useful; the chapter is the
    smallest meaningful unit.
  * System-tagged books are skipped (auditing
    the Threads book against itself is a
    no-op).

* **Blind-spots pre-pass** — computed BEFORE the
  LLM ever runs so the model has hard evidence
  rather than guessing from prose alone.  Three
  detectors:
  * **Dormant in scope** — thread has links
    elsewhere in the project but zero in the
    cursor scope.  Flags arcs the section is
    neglecting.
  * **Payoff unfired** — thread status is
    `payoff` but no paragraph in the project
    links to it.  Surfaces declared payoffs the
    author hasn't actually written yet.
  * **Zero links** — thread status is past
    `setup` but the project contains zero
    references.  Catches the "I declared this
    thread, but never followed through" gap.

* **Prompt envelope** composer reads:
  * Every Thread paragraph's full HJSON body
    (title / status / weight / arc shape /
    connections / tension / register / notes).
  * The blind-spots list with the kind label
    for each.
  * Every paragraph in the scope (title + body +
    outgoing-link count).
  * A system prompt that asks for four-part
    output:
    1. Per-paragraph thread scoring (advances /
       touches incidentally / should advance
       but doesn't).
    2. Confirm or refute each blind spot.
    3. Flag threads whose declared status looks
       wrong given the manuscript evidence
       (e.g. `setup` with many incidental
       touches; `payoff` without the payoff
       landed).
    4. One-paragraph summary of the scope's
       narrative health.

* **Streaming** — runs through the shared
  `spawn_chat_stream` + `Inference` machinery
  (same plumbing every other AI flow uses).
  Focus jumps to the AI pane; the response
  streams in.  Standard `I` / `T` / `B` apply
  chords work for selectively pulling sections
  of the audit into a Notes paragraph.

* **No new tests** — Phase A.3 is integration
  glue (HJSON parsing covered by Phase A.1;
  hierarchy walking covered by existing tree
  tests; envelope composition has no isolatable
  pure-logic surface).  576 tests still pass.

* **Implementation surfaces:**
  * `Action::AiThreadAudit` — rebindable via
    `keys.bindings`.
  * `App::start_thread_audit` +
    `resolve_thread_audit_scope` +
    `collect_scope_paragraphs` +
    `compute_blind_spots` +
    `collect_thread_bodies_full` +
    `compose_thread_audit_prompt` in
    `src/tui/app/threads_impl.rs`.
  * `ScopedParagraph` + `BlindSpot` /
    `BlindSpotKind` private types in the same
    module.
