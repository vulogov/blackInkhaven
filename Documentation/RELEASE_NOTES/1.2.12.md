# Inkhaven 1.2.12 — in flight

1.2.11 was the "multilingual prompts" release — three
phases of resolver work plus the AI sentence-rhythm
rewrite, concordance Enter-jump, AI-diff wrapping, the
two bootstrap CLIs (SDT + prompts), and the config-TUI
polish trio (F3 file-picker / new enums / HSL slider).
1.2.12 picks up the carryovers that didn't fit and the
new ground 1.2.11 surfaced.

This document is a living scratchpad — entries land
here as the work lands.  Each item starts as a one-
line plan; gets fleshed out when the corresponding
feature ships; gets a chord / config / hook callout
when relevant.

## Planned themes

### Carryovers from 1.2.11

  * **Apply prompts-editor critique back into the
    main TUI's flows.**  The prompts-editor's
    reviewer-LLM is the deepest review surface we
    have; the main TUI's `Ctrl+B 0` HJSON editor
    is the most exposed.  Wiring the same
    review-as-you-edit shape into the in-app
    HJSON editor would close the loop.

  * **Build-time doc-comment extraction polish.**
    The current extractor handles `Option<T>` /
    `Vec<T>` / `HashMap<K, V>` descents; nested
    `Option<HashMap<K, V>>` shapes still don't
    flow through cleanly.  Audit + fix.

  * **Concordance export — `inkhaven export
    concordance`.**  CSV / JSON variants of the
    Ctrl+B Shift+L modal data for use in
    spreadsheets / analysis pipelines.

  * **Mouse-driven scene-break selection.**  Drag
    from one break-line to the next selects the
    scene.  Pairs with the existing Ctrl+B </>
    scene-break navigation.

  * **Show-don't-tell theme audit.**  Per-detector
    underline style override (DIM / BOLD /
    REVERSED / NONE), since the teal underline
    reads differently on different terminal
    palettes.

## Implementation log

### Split view — Phase A (foundation)

Plumbing for the fullscreen two-paragraph split-view
layout.  No user-visible change beyond a new
`Shift+F4` chord that toggles a flag.  Three new
state pieces: `App.split_view: bool` (the layout
flag), `App.picker_target: PaneTarget` (where the
next picker pick lands — defaults to Primary; Phase
B wires the routing), and a renderer dispatch
branch that switches to a 50/50 horizontal editor
split when `split_view` is true AND
`secondary.is_some()`.  Existing `F4` (same-
paragraph snapshot split) and `Ctrl+F4` (accept
snapshot) are untouched — the new chord is
additive.  `pin_secondary_by_uuid` and
`drop_secondary` helpers exist as Phase B targets.
Two new tests lock the binding contract
(`existing_f4_and_ctrl_f4_bindings_remain_untouched`,
`pane_target_defaults_to_primary`).

### Split view — Phase B (Shift+Enter pins to secondary)

The user-visible payoff.  Every paragraph picker
now routes a `Shift+Enter` accept to the split-
view secondary slot instead of replacing the
primary:

  * Tree-pane Enter (same paragraph node the tree
    is currently selected on).
  * Fuzzy paragraph picker (`Ctrl+V P`).
  * Recent-paragraph picker (`Ctrl+V Shift+P`) —
    shares the fuzzy picker's modal shape, so wiring
    one covered both.
  * Bookmark picker (`Ctrl+V M`).

The dispatch goes through a new
`App::dispatch_picker_accept(id,
pin_to_secondary: bool)` helper that surfaces
pin failures on the status bar (e.g. trying to pin
the paragraph that's already in the primary slot).
Picker footer hints + opening status echoes
updated to advertise the new chord.  One new test
(`shift_modifier_drives_pin_to_secondary`) locks
the contract that the SHIFT modifier flag is what
drives the routing.

The natural workflow: `Ctrl+V P` (or `Ctrl+V M`),
filter to the paragraph you want beside the
current one, `Shift+Enter`, then `Shift+F4` to
view the split.

499 tests pass (was 498 — +1).  Zero new
dependencies.  Phase C adds per-pane chrome
(focus highlights, Tab swap behaviour); Phase D
adds the translation-workflow sibling-book picker.
