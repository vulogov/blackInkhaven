# Inkhaven 1.2.11 — in flight

1.2.10 was the "standalone editors" release —
`inkhaven config` and `inkhaven prompts-editor` both
shipped a dedicated TUI for their respective HJSON
files.  1.2.11 picks up the carryovers from 1.2.10
and the deferred items from 1.2.9.

This document is a living scratchpad — entries land
here as the work lands.  Each item starts as a one-
line plan; gets fleshed out when the corresponding
feature ships; gets a chord / config / hook callout
when relevant.

## Implementation log

### A1 — AI sentence-rhythm rewrite (`Ctrl+B Shift+M`)

Pairs with the existing `Ctrl+B Shift+H`
sentence-rhythm gauge.  Where the gauge says
MONOTONE, the rewrite asks the configured LLM to
mix short and long sentences while preserving the
author's voice, tone, named characters, dialogue,
em-dashes, and paragraph breaks.

Prompt resolution follows the standard inkhaven
pattern (`grammar-check` style precedence):

  1. Paragraph in the project's `Prompts` system
     book with slug or title
     `sentence-rhythm-rewrite` /
     `sentence rhythm rewrite`.
  2. Entry in the project's `prompts.hjson`
     library.
  3. Embedded multilingual fallback —
     `sentence_rhythm_rewrite_default_prompt(language)`
     in `src/tui/app.rs`.  Language-aware: the
     prompt explicitly names the project's
     `language` field so the rewrite lands in
     the same language (the model wouldn't
     translate but for the strong directive).

Behaviour:

  * `Ctrl+B Shift+M` in editor scope spawns the
    inference; the streaming response goes into
    the AI pane (the user can watch it land in
    real time).
  * The chord ALSO fires from inside the
    `Ctrl+B Shift+H` rhythm-gauge modal — the
    natural diagnose-then-rewrite path: open the
    gauge, see MONOTONE, press the chord, watch
    the rewrite stream.  The gauge dismisses
    automatically as the inference spawns
    (otherwise it would sit stuck on a stale
    verdict while the AI pane lit up behind it).
    Implemented two-part: (a) the rhythm-modal
    handler intercepts the meta-prefix
    (`Ctrl+B`) and arms `meta_pending` after
    closing the modal, so the normal meta
    dispatch resolves `Shift+M` →
    `AiRewriteRhythm` on the next key; (b)
    `start_sentence_rhythm_rewrite` also
    defensively dismisses any lingering
    `Modal::SentenceRhythm` so the rewrite UX
    is coherent regardless of entry point.
    Generalises: any meta-action is now
    reachable from inside the gauge.
  * When the stream completes, an AI diff modal
    pops **automatically** — the user doesn't
    need to press a chord to trigger review.
    Side-by-side: original on the left,
    rewrite on the right, ↑↓ / PgUp / PgDn /
    Home / End scroll.
  * `a` / `Enter` accept: a snapshot of the
    pre-rewrite buffer is created with
    annotation `Sentence rhythm rewrite` (so
    F6 surfaces it as a discoverable history
    entry), THEN the buffer is replaced.
  * `r` reject: modal closes, buffer untouched.

Implementation notes:

  * New `App.pending_rhythm_rewrite: bool` flag
    set by `start_sentence_rhythm_rewrite` and
    cleared by `pump_inference` on transition
    to Done (or Error).  When set + Done +
    non-empty response, `pump_inference`
    auto-opens the diff modal via the new
    `open_ai_diff_review_with_snapshot` helper.
  * `Modal::AiDiffReview` gains a
    `post_accept_snapshot: Option<String>`
    field.  When `Some`, the accept handler
    (in `ai_diff_review_handle_key`) calls
    the new `snapshot_open_paragraph_with_annotation`
    helper BEFORE `apply_ai_diff_accepted` so
    the pre-rewrite state is preserved.
  * The snapshot helper lives in
    `src/tui/app/snapshot_impl.rs` next to
    `commit_snapshot_annotation` and reuses the
    same `store.create_snapshot_annotated` API
    F5 already uses.

**AI diff pane wraps long lines.**  Until now the
side-by-side diff truncated content past the column
boundary — a single long sentence on either side
became unreadable.  The renderer now wraps each
diff row at column width, prepending the diff
marker (`- ` / `+ ` / `  `) on the first wrapped
row and indenting continuation rows by two columns
so a wrapped sentence reads as one visually
continuous block.  Left and right wrap
independently, but the shorter side is padded with
empty rows so paired diff entries stay vertically
aligned — the side-by-side guarantee is preserved.
Implementation: new `wrap_diff_row` helper in
`src/tui/app/render/modals.rs`; new
`Modal::AiDiffReview.wrapped_total` field written by
the renderer each frame and read by the key handler
to clamp Down / PgDn / End against the post-wrap
row count.  Scroll counts wrapped rows, not source
diff lines.

460 tests still pass.  Zero build warnings.

Docs:

  * `Documentation/KEYBINDING.md` row 86 added.
  * Action description in `src/tui/keybind.rs`
    covers prompt-resolution + accept/reject
    semantics + the snapshot annotation.

### Concordance: jump-to-occurrence on Enter

Carryover from 1.2.10.  The `Ctrl+B Shift+L`
concordance modal was read-only — users could see
that the word `walk` occurs 47 times across the
project but had no way to land in the editor on
any of them.  Now: pressing **Enter** on the
selected row closes the modal and navigates to the
first sample's source paragraph at the
corresponding editor line.

Implementation notes:

  * `concordance_jump_to_selection` in
    `src/tui/app.rs`.  Pulls `(slug_path, line_no,
    headword)` out of the first sample of the
    selected entry, resolves the slug path to a
    paragraph node via
    `Hierarchy::find_by_path` (exact inverse of
    `slug_path` used at index-build time), opens
    via the existing `open_paragraph_by_uuid`,
    then jumps the textarea cursor via
    `tui_textarea::CursorMove::Jump`.
  * **Heading-offset translation.**  The
    concordance index is built over
    *heading-stripped* paragraph bodies (so
    `line_no` is 1-based within the prose).  The
    editor displays the *raw* paragraph
    (including the `= title` heading line and
    any blank lines after it).  Naively jumping
    to row `line_no - 1` would land on the
    wrong line.  The jump function replays the
    same heading-strip logic on the live
    textarea, computes the prefix offset, and
    adds it to `line_no - 1` so the cursor
    lands on the right row regardless of the
    paragraph's heading shape.
  * **Graceful failure modes.**  Source
    paragraph no longer exists (rename / delete
    after index was built) — status reports
    `concordance: source paragraph
    <slug_path> no longer exists`, modal stays
    open.  Selected entry has zero samples
    (shouldn't happen — index dedup keeps at
    least one — but defensive) — status
    reports the empty case, modal stays open.
    Open-paragraph failure — status surfaces
    the underlying error, modal already
    closed.
  * Plain `Enter` only; `Ctrl+Enter` and
    `Alt+Enter` fall through to the filter
    input (which today no-ops them, the
    desired behaviour).
  * **System books excluded from the corpus.**
    `open_concordance` now skips any paragraph
    whose ancestor chain contains a node with
    `system_tag.is_some()` — Prompts,
    Characters, Places, Lore, Help, Notes,
    Artefacts, Typst, Scripts.  Two reasons:
    (1) those books are metadata /
    scaffolding, not prose, so they dilute
    the lexical signal (the keybind
    description has always promised "the
    words actually carrying the prose's
    weight").  (2) System-book paragraphs
    are stored only in bdslib (the prompts
    editor never writes `.typ` files), so an
    Enter-jump to one would fail with
    `read …: No such file or directory`
    inside `load_paragraph`.  Filtering them
    out at index time eliminates the broken
    nav case entirely and tightens the
    concordance to actual manuscript prose.

### Show-don't-tell: per-language built-ins + `bootstrap` CLI

Carryover from 1.2.9.  Until now the show-don't-tell
overlay (`editor.style_warnings.show_dont_tell`)
only had built-in word lists for English; Russian,
French, German, and Spanish projects saw nothing
until the user filled in HJSON by hand.  Two
changes land together:

**Built-in lists for RU/FR/DE/ES.**
`built_in_linking_verbs` / `built_in_emotion_adjectives`
/ `built_in_manner_adverbs` / `built_in_cognition_verbs`
in `src/config.rs` now return curated lemmas for
all four languages.  Conservative + dictionary-
shape — the major emotion families (anger /
sadness / fear / joy / fatigue / surprise / shame)
plus the obvious copula / cognition verbs.  Per-
genre or per-corpus tuning is the LLM bootstrap's
job (below), not the built-ins'.  Snowball
stemming at match time means a handful of canonical
forms cover the inflection space.

**`inkhaven show-dont-tell bootstrap <lang>`.**
New CLI subcommand that uses the configured LLM as
a one-shot vocabulary curator.  Reads
`inkhaven.hjson` for the LLM provider, asks for the
four lists in the requested language, parses the
JSON response, and emits a copy-paste-ready HJSON
snippet on stdout.  **Never writes to
`inkhaven.hjson` automatically** — mirrors
`doctor --filter-words-snippet`: the author
reviews the snippet and pastes what they like.

Implementation notes:

  * `src/cli/show_dont_tell.rs`.  Streams the
    response (dots to stderr for progress so the
    HJSON on stdout stays pipe-clean), strips
    optional markdown code fences, brace-balanced
    JSON extraction so stray model chatter
    doesn't kill the parse.  Four unit tests lock
    the parser against the obvious failure modes
    (chatter wrapping, nested braces in string
    values, fenced output, plain garbage).
  * **Strict JSON contract.**  The system prompt
    instructs the model to emit ONE JSON object
    with four arrays and no other text;
    `extract_json_object` walks a brace-balanced
    range so an example object embedded in
    explanatory prose still parses correctly.  On
    parse failure the raw response is dumped on
    stderr and the command exits non-zero —
    debug + retry is one keypress.
  * **Optional `--genre` hint.**  Folded into the
    user prompt so the model picks vocabulary at
    the right register ("literary fiction",
    "thriller", "YA fantasy", …).  Empty by
    default — the genre-neutral built-ins ship
    so nobody *has* to run this.
  * **Why not always-on AI detection?**  Considered
    and rejected.  The overlay updates as the
    user types; running an LLM per keystroke is
    fatal on latency (~1-3s per call), expensive
    on tokens, and breaks offline use — a core
    value of a terminal editor.  Using the LLM
    as a one-shot curator gives the multilingual
    coverage without paying the always-on cost,
    and the existing `Ctrl+B Shift+T` AI
    show-don't-tell scan remains the deep-
    analysis layer for context-sensitive cases.
  * Test `sdt_unsupported_language_falls_back_quiet`
    updated to use `"klingon"` (or any other
    language inkhaven doesn't ship lists for)
    since German now has built-ins — the
    semantic the test locks (empty config +
    no built-ins → `is_empty()` detector) is
    preserved.
